# v1.4.0 与 v1.5.0 大众化发布文案设计

## 1. 目标

把已经发布的 `v1.4.0` 与 `v1.5.0` 从“内部交付记录”改写为普通玩家能够快速理解的游戏更新公告，同时保留必要的版本边界、历史事实和审计信息。

本轮只调整公开叙事和文档，不改变标签目标、版本内容、游戏代码、在线 Demo、发布产物或任何开发分支状态。

## 2. 文案原则

- 标题先表达玩家能够理解的主题，不使用 `UI Foundation`、`压力模型`、`facade`、`release-prep` 等内部术语作为主标题或开场重点；
- 正文第一段回答“这次更新带来了什么体验”，随后列出玩家可感知的变化；
- 技术实现、精确提交、测试和历史元数据下沉到末尾的“开发与版本说明”；
- 不把基础架构宣传为新美术，不把规则变更宣传为已部署的在线 Demo，也不把独立分支内容写成正式版本内容；
- 中文和英文 README 保持事实一致，但英文使用自然对应名称，不逐字硬译中文宣传语。

## 3. 版本定位与标题

### v1.4.0

- 中文公开标题：`v1.4.0 — 收容行动整备`
- 英文对应名称：`v1.4.0 — Containment Readiness`
- 定位：面向启动、资源兜底和重复开局稳定性的回溯更新，不是画面重制或新玩法版本。

公开摘要：

> 这是一次面向稳定性的回溯更新，不是画面重制。游戏的启动、资源加载和重新开始流程得到整理，让连续多局游玩更加可靠，也为以后逐步加入正式美术和音频素材做好准备。

玩家正文使用以下要点：

- 修复连续重新开始后可能累积的后台监听，降低多局游玩后出现异常的风险；
- 统一游戏启动和资源加载流程；素材缺失时仍使用程序化画面和合成音效；
- 现有武器、敌人、HUD、音效和六分钟流程保持不变。

开发与版本说明必须保留：

- 这是对 2026-07-10 已完成内容的回溯记录；
- `v1.4.0` 标签仍固定指向 `2f9c28534d6b99f0f0996f44bb236ecb235e8d4f`；
- 不重写原提交，因此其中的历史包版本号保持原样；
- 不包含正式 UI/美术素材、SCP-049 新终局战或 Windows 客户端，也不附加声称来自独立 v1.4.0 构建流程的二进制产物。

### v1.5.0

- 中文公开标题：`v1.5.0 — SCP-049：疫医狂潮`
- 英文对应名称：`v1.5.0 — SCP-049: Surgical Frenzy`
- 定位：突出狂热阶段、弱点窗口、混合增援与敌群复制带来的终局决策，不以内部“压力模型”命名。

公开摘要：

> 六分钟生存战的终点现在更加危险。SCP-049 会在战斗中进入“外科狂热”：它停止追击、召来混合增援，同时暴露短暂的弱点。玩家必须在清理不断复制的敌潮和集中攻击 Boss 之间迅速选择。

玩家正文使用以下要点：

- SCP-049 终局战加入普通、外科狂热和濒死阶段，不再只是单纯追击和召唤；
- 狂热阶段出现短暂弱点窗口，此时 SCP-049 承受更多伤害；
- 防暴镇压单位、闪现潜行者、复制生物体和安保无人机可能同时加入围攻；
- 非 Boss 敌人会尝试自我复制，拖延清场会让局势迅速失控；
- 敌人总量设有安全上限，避免复制与召唤无限增长。

开发与版本说明必须保留：

- 狂热弱点倍率、复制间隔、230 个活动敌人上限以及收容突破器伤害上限等精确规则；
- 11 个 Node 回归测试以及 CI 先测试、后构建的质量门禁；
- 功能边界为 `daeb3ed605f29a2d97a30458ee12109561dbaf6d`，标签仍固定指向 `c92da05d8c8c066c462b08c3b6db21bee877d31b`；
- UI/Art、角色动画、橘色角色路线与客户端平台均未随本版本发布；
- 在线 Demo 尚未核对为本标签，不能声称已经部署 `v1.5.0`。

## 4. 页面与文档结构

GitHub Release 页面按以下顺序展示：

1. GitHub Release 名称；
2. 一段玩家摘要；
3. “本次改善”或“主要更新”；
4. 折叠的“开发与版本说明”。

仓库中的 `docs/releases/v1.4.0.md` 与 `docs/releases/v1.5.0.md` 保留一级标题，作为完整文案源。同步 GitHub Release 时移除正文中的首个一级标题和紧随其后的空行，避免页面标题重复；其余正文保持一致。

README 只用一至两句说明最近版本和玩家可感知主题，不在顶部堆叠提交、分支和内部接口。CHANGELOG 继续承担较精确的规则与工程记录，但版本标题和开场摘要与公开 Release 一致。

## 5. 修改范围

主要文案文件：

- `docs/releases/v1.4.0.md`；
- `docs/releases/v1.5.0.md`；
- `README.md`；
- `README.en.md`；
- `CHANGELOG.md`。

为避免公开文档继续显示旧名称，还需要最小同步：

- `docs/project-status.md` 中的正式版本与回溯里程碑名称；
- `docs/README.md` 中的两条 Release 链接标题。

外部写入范围仅包括：

- 修改 GitHub `v1.4.0` Release 的名称和正文；
- 修改 GitHub `v1.5.0` Release 的名称和正文；
- 将上述文档提交以普通快进方式推送到 `main`。

明确不修改：标签对象与目标提交、版本号、源代码、玩法配置、测试、UI/Art 或 App Platform 分支、在线 Demo、Release 附件、Latest 归属和历史提交。

## 6. 同步顺序与失败处理

1. 先完成并验证仓库文档修改；
2. 创建单一文档提交；
3. push 前重新 fetch，确认远端 `main` 未漂移且可以普通快进；
4. 推送 `main`；
5. 逐一更新两个现有 GitHub Release；
6. 从 GitHub API 反查标题、正文、草稿状态、预发布状态、Latest 状态和标签目标。

如果仓库提交或 push 失败，不更新 GitHub Release。如果 `v1.4.0` 更新成功而 `v1.5.0` 更新失败，保留已成功更新的页面，不执行删除、重建或标签改写；修复失败原因后重试未完成的页面，并明确报告部分完成状态。

## 7. 验证标准

- 旧公开标题 `UI Foundation（回溯里程碑）` 和 `SCP-049 压力模型` 不再出现在当前公开版本入口；冻结接口或历史设计语境中的 `UI Foundation` 不做机械替换；
- README 中文、英文版本状态与 CHANGELOG、Release 源文件一致；
- v1.4.0 没有被描述为画面重制、新素材或新玩法；
- v1.5.0 没有包含 UI/Art、客户端或未验证 Demo 的发布声明；
- Markdown 本地链接检查、`git diff --check`、`node --test` 与 `npm run build` 通过；
- GitHub 两个 Release 均保持非草稿、非预发布，`v1.5.0` 继续是 Latest；
- 远端 peeled tag 仍分别为 `2f9c28534d6b99f0f0996f44bb236ecb235e8d4f` 与 `c92da05d8c8c066c462b08c3b6db21bee877d31b`；
- GitHub Release 正文与对应源文件在移除一级标题、规范化换行后逐字一致；
- 最终 `main` 与 `origin/main` 同步，工作树干净。
