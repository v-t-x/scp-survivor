# SCP Survivor 多平台客户端设计

**日期：** 2026-07-14
**状态：** 已批准进入实施计划阶段
**批准范围：** Windows 桌面测试客户端、Android 测试 APK、iOS/iPadOS 测试 App、微信小游戏技术验证与测试版本
**不自动授权：** 正式上架、公开发布、签名证书采购、自动更新、商店接入、付费服务或商业发行

## 1. 目标

在不复制玩法代码、不改变游戏语义的前提下，让同一套 Phaser + Vite 游戏核心可以被以下客户端承载：

- Windows 桌面客户端；
- Android APK；
- iOS/iPadOS App；
- 微信小游戏。

第一轮目标是获得可验证的工程测试版本，不是立即制作正式发行包。UI/Art 完成前可以验证平台外壳、输入、存储、音频、生命周期和资源合同；最终素材合入后必须重新执行完整验证。

## 2. 选定路线

采用“一套游戏核心 + 三类平台适配”的架构：

| 目标 | 首选路线 | 责任边界 |
|---|---|---|
| Windows | Electron 桌面容器 | 承载标准 Web 构建，处理窗口、进程、离线、安全和安装 |
| Android | Capacitor Android | 承载标准 Web 构建，适配触摸、横屏、系统返回、暂停恢复和移动音频 |
| iOS/iPadOS | Capacitor iOS | 与 Android 共用移动端合同，适配安全区域、后台恢复、签名和 TestFlight |
| 微信 | 微信小游戏独立适配层 | 适配小游戏运行时、Canvas、存储、音频、触摸、资源和包体限制 |

Electron 不进入游戏 renderer，Capacitor 插件不成为玩法依赖，微信 API 不直接散落在游戏系统中。平台差异通过明确的 adapter/capability 边界处理。

## 3. 未选路线

### 3.1 先完成三个客户端，最后再研究微信

这条路线可以更快得到 Windows 和移动端外壳，但会把技术差异最大的微信小游戏风险推迟到最后，可能导致公共接口返工，因此不采用。

### 3.2 为移动端或微信重写一套游戏

这会形成多套玩法实现，增加平衡、Bug 修复和内容同步成本，也违背继续使用 Phaser、Vite 和共享游戏语义的既定方向，因此不采用。

## 4. 架构边界

### 4.1 游戏核心

游戏核心继续负责：

- 玩法、敌人、武器、Boss、事件、时间轴和胜负；
- Phaser Scene、Preload、manifest 和 fallback；
- UI 状态和游戏内音频语义；
- `scp-survivor-meta` 数据结构及兼容规则。

平台层不得通过不同伤害、速度、刷新率、升级概率或胜负条件来掩盖设备差异。

### 4.2 平台层

平台层负责：

- 启动、退出、暂停、恢复和异常终止；
- 窗口、横竖屏、安全区域和系统返回；
- 平台存储、音频解锁、权限和离线行为；
- 构建产物 staging、打包、安装和测试分发；
- 平台安全默认值及运行时限制。

### 4.3 拟议目录

具体路径在实施计划中确认，优先保持以下隔离方式：

```text
clients/
  desktop-electron/
  mobile-capacitor/
  wechat-minigame/
```

各客户端拥有自己的依赖、生成目录和忽略规则。除非验证证明必要，不修改根依赖、Web 入口、Vite 配置或游戏源码。

## 5. 必须统一的平台合同

### 5.1 构建合同

- Windows、Android 和 iOS 优先消费根项目生成的标准 `dist/`；
- 客户端不得从 `src/` 直接启动生产版本，也不得依赖 Vite dev server；
- staging 过程记录游戏 commit、dirty 状态、文件清单和 hash；
- 微信小游戏可能不能原样消费 HTML 构建，技术验证阶段只允许建立可重复的转换或专用构建流程，不复制玩法源代码；
- 对 Vite `base`、入口或资源路径的修改必须由真实失败证据触发，并单独批准。

### 5.2 输入合同

公共输入层表达游戏动作，而不是具体按键或平台 API：

- 移动向量；
- 确认、取消和暂停；
- 静音及已批准的辅助面板操作；
- 调试操作只存在于开发构建。

Windows 保留键鼠。Android、iOS 和微信共用触摸动作语义。触摸控件的功能和可测试区域先确定，最终皮肤、动画和布局由 UI/Art 里程碑决定。

### 5.3 存储合同

- Web、Electron 和 Capacitor 首先保持现有 `scp-survivor-meta` schema；
- 微信小游戏通过存储 adapter 保存同一逻辑 schema；
- 读取失败时保持当前安全回退；
- 更换 origin、app id、bundle id、小游戏标识或 user-data 路径前必须提供迁移方案；
- 第一轮不实现账号、云存档或跨设备同步。

### 5.4 音频与生命周期合同

- 首次明确用户交互后允许音频恢复；
- 暂停、切后台、恢复和 Scene restart 不得造成叠音或 AudioContext 泄漏；
- 平台最小化或后台事件不自行改变游戏计时语义；
- 如需暂停策略，由共享合同明确后再修改游戏层。

### 5.5 资源合同

- manifest、Preload 和 fallback 继续是资源入口；
- 资源 key、frame 结构、大小写和相对路径必须可验证；
- UI/Art 合入前以 fallback 和少量测试资源验证流程；
- UI/Art 合入后重新验证图片、atlas、spritesheet、字体、采样音频、包体和显存；
- 客户端不得绕过素材来源与许可证登记。

## 6. 实施阶段

### 阶段 0：计划与环境门禁

- 保留并审查 `dev/app-platform` worktree 中现有未跟踪提案；
- 在不丢失提案的前提下，将 App 分支同步到批准的 `main` 基线；
- 锁定 Windows、Android、iOS 和微信测试目标、工具版本与目录边界；
- 检查 Android SDK/设备、微信开发者工具，以及 iOS 所需 Mac、Xcode、设备和账号是否可用；
- 对需要共享文件变更的接口逐项申请批准。

### 阶段 1：Windows 工程 PoC

- 使用 Electron 承载标准 Web artifact；
- 验证单窗口、固定安全 origin、完全退出、离线、存档、音频、DPI 和资源路径；
- 第一轮只要求 unpacked 内部测试版本；是否生成 unsigned installer 由实施计划单独决定。

Windows 是最低风险起点，用于先建立可复用的构建、staging 和 smoke 流程。

### 阶段 2：微信小游戏可行性验证

- 验证 Phaser 启动、Canvas/WebGL 和帧率；
- 验证触摸输入、小游戏存储、音频解锁和生命周期；
- 验证资源加载、包体和分包边界；
- 最小成果是能够进入游戏、移动、开始一局、保存并重新读取测试进度；
- 如果 Phaser 或关键插件不兼容，先提交证据和替代方案，不直接重写游戏。

微信阶段提前进行，因为它与标准浏览器/WebView 的差异最大。

### 阶段 3：Android 测试 APK

- 使用 Capacitor 创建隔离的 Android 客户端；
- 生成可侧载的测试 APK；
- 验证横屏、触摸、返回键、暂停恢复、音频、存档、离线和性能；
- 至少在一台真实 Android 设备上完成 smoke；
- Play Store、AAB 和正式签名不属于第一轮。

### 阶段 4：iOS/iPadOS 测试 App

- 复用移动端公共合同并创建 Capacitor iOS 客户端；
- 在 Mac/Xcode 环境验证 simulator 和真机；
- 验证横屏、安全区域、触摸、音频、存档、后台恢复和性能；
- TestFlight、正式签名和 App Store 提交需要单独授权与账号门禁。

### 阶段 5：UI/Art 后的最终集成验证

- 基于玩法与 UI/Art 均已合入的 `main` 重建全部客户端；
- 重新验证正式资源、画面比例、触摸布局、音频、包体、启动时间和内存；
- 检查图标、启动图、名称、版本、许可证和署名；
- 只有全部目标通过各自验证后，才另行制定正式发布计划。

## 7. 验证模型

所有平台都必须先通过 Web production build。各平台额外验证如下：

| 验证项 | Windows | Android | iOS | 微信小游戏 |
|---|---:|---:|---:|---:|
| 启动与完全退出 | 必须 | 必须 | 必须 | 必须 |
| 离线运行 | 必须 | 必须 | 必须 | 按小游戏资源策略验证 |
| 输入 | 键鼠 | 触摸/系统返回 | 触摸/安全区域 | 触摸 |
| 存档与损坏回退 | 必须 | 必须 | 必须 | 必须 |
| 首次交互音频 | 必须 | 必须 | 必须 | 必须 |
| 暂停/恢复 | 窗口最小化 | 前后台 | 前后台 | 小游戏生命周期 |
| 真机测试 | Windows 10/11 | 至少一台 Android | 至少一台 iPhone/iPad | 微信开发者工具及真机 |
| 安全与权限 | Electron 隔离 | Android 最小权限 | iOS 最小权限 | 微信能力最小化 |

每个平台未执行的项目必须标为“未测试”，不能从其他平台的通过结果推断。

## 8. 风险与门禁

1. **触摸输入风险：** 当前游戏主要按键鼠设计。平台层可以定义动作合同，但最终触摸布局和可玩性必须与 UI/Art 协调。
2. **微信运行时风险：** 微信小游戏不是普通浏览器或 Capacitor WebView，可能需要专用构建和 API adapter，必须先做小型技术验证。
3. **iOS 环境风险：** 最终构建、签名和设备测试依赖可用的 Mac、Xcode、开发者账号和证书。
4. **资源风险：** fallback 通过不代表正式图片、atlas、字体和音频已经通过客户端验证。
5. **身份与存档风险：** origin、app id、bundle id 或小游戏标识变化可能让旧存档看似丢失，进入测试数据后应尽早冻结。
6. **许可与发布风险：** SCP 衍生内容、正式素材、字体、音频和平台依赖必须在公开发布前复核许可与署名。
7. **维护风险：** Electron、Capacitor、移动 SDK、Xcode 和微信工具都需要持续升级；版本锁定和升级节奏在各平台实施计划中定义。

## 9. 明确排除

本设计不包含：

- macOS 或 Linux 桌面客户端；
- Steam、Microsoft Store、Google Play、App Store 或微信正式发布；
- 自动更新、Launcher、差分下载、账号、云存档和跨设备同步；
- 广告、内购、付费、排行榜、社交分享或平台成就；
- 遥测、崩溃上报或远程配置；
- 为任何平台改变核心玩法、平衡、胜负或存档语义；
- 在 UI/Art 完成前确定最终触摸视觉、图标和启动品牌素材。

## 10. 下一步授权边界

本设计批准后，下一步只编写详细实施计划。实施计划必须：

- 把 Windows、微信、Android 和 iOS 拆成可独立验证的阶段；
- 先列出依赖和环境门禁，再允许添加脚手架；
- 明确所有共享文件修改申请；
- 为每个平台定义回滚与最终 Git 状态检查；
- 在大范围或难回退的技术栈实施前按 Level 2 进行外部 GPT 审查。

在实施计划获得批准前，不添加平台依赖、不创建客户端脚手架、不生成安装包或商店产物。
