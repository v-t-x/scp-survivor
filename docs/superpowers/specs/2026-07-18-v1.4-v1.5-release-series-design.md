# v1.4.0 与 v1.5.0 发布序列设计

## 1. 目标

在不重写历史的前提下，为 `v1.3.0` 之后已经完成的主线工作恢复有意义的版本边界，并把当前 `main` 整理成内容、版本元数据和公开文档一致的 `v1.5.0` 发布候选。

本轮只准备 `v1.4.0` 与 `v1.5.0`。标签、GitHub Release 和 push 必须在最终 diff、验证结果与目标提交再次展示后，取得项目所有者的单独确认才可执行。

## 2. 版本划分原则

- minor 版本必须对应独立、可说明、可验证的产品或工程能力，不按提交数量和日期机械切分；
- 纯治理、计划、文档整理和 CI 调整不单独占用 minor 版本号；
- 已经被后续回滚的实现不形成版本；
- 未进入稳定主线、未完成项目所有者验收的开发分支中间状态不伪装成正式稳定版本；
- `v2.0.0` 留给核心循环、存档兼容、产品结构或正式平台形态发生代际变化的版本。

## 6. 当前实施范围

本轮允许修改：版本元数据、发布模式开关、CHANGELOG、README、当前状态与设计事实、发布说明草稿，以及本设计和实施计划。

README、CHANGELOG 与状态文档按发布后的最终公开状态准备，但只能保留在隔离发布分支；在标签、GitHub Release 与 main push 获得再次授权并能作为同一发布操作完成之前，不得把这些最终态文案推到公开 `main`。

本轮禁止：创建或推送标签、创建 GitHub Release、push、merge、rebase、squash、重写历史、修改 UI/Art 或 App Platform 分支、实现角色动画路线、部署在线 Demo。

## 7. 发布门禁

- 根版本元数据一致：`package.json`、`package-lock.json` 均为 `1.5.0`；
- 正式构建关闭调试入口：`DEBUG_MODE=false`，文档不再声称 `B` 跳关可用；
- `node --test` 全部通过；
- `npm run build` 成功；
- Markdown 本地链接检查通过；
- `git diff --check` 无错误；
- worktree 除已批准文件外无未知改动；
- README、CHANGELOG、标签与 GitHub Release 必须指向同一 `v1.5.0` commit；在线 Demo 只有在被明确声明为该版本时才必须指向同一 commit，否则必须标记为未核对或独立部署；
- 创建标签和 Release 前重新 fetch，确认远端 ancestry、目标 commit 与 GitHub 现有标签/Release 状态未变化。
