# UI/Art 正式素材垂直切片设计

## 1. 状态与决策权

- **状态：**项目所有者已于 2026-07-11 批准本设计方向。
- **负责分支：**`feature/ui-art-overhaul`。
- **目标：**让 SCP Survivor 从程序化几何原型跨越到具有正式角色、环境、界面、光影和反馈的可识别游戏画面。
- **产品决定：**采用“工业恐怖 × 战术生存”的视觉方向，以精细像素世界、高清基金会 UI 和动态光影组成最终画面。
- **技术边界：**继续使用 Phaser 3、Vite、JavaScript ES Modules、现有 `PreloadScene`、asset manifest 和 fallback 合同。
- **授权边界：**本文件批准设计，不直接批准代码或素材提交；实施必须先形成书面计划并经过正常审查、测试和素材准入。

本方向属于整体视觉升级。项目所有者已经直接确认方向，因此不需要重复请求外部 GPT 决定风格；正式素材的许可、商业使用和来源仍必须逐项验证。

## 2. 当前问题

第一轮 UI 改造主要完成 Theme token、标题页、武器选择和 HUD 的基础迁移。由于该批次明确禁止引入图片、图标、背景、动画和正式素材，并要求坐标、尺寸和结构基本不变，所以最终变化集中在颜色、字体、边框、状态标签和低生命提示。

这批工作建立了兼容基础，但没有解决成品感问题：

- 玩家、普通敌人、精英和 SCP-049 仍由简单几何纹理表示；
- 地面、墙体、门、设备和设施装饰缺少正式素材；
- UI 缺少图标、材质、组件造型和完整覆盖层统一；
- 环境光、警报、污染、血迹和异常失控缺少视觉层次；
- 程序化 fallback 承担了正式画面的职责，导致游戏仍像功能原型。

下一阶段不再以“继续替换 token”为主要任务，而是建立首个正式素材垂直切片。

## 3. 视觉目标

### 3.1 核心组合

- **世界与角色：**现代精细像素画；
- **界面：**高清、克制、专业的基金会设施 UI；
- **氛围：**煤黑、脏灰、冷白、琥珀应急灯和克制红色警报；
- **动态表现：**Phaser 驱动的光照、阴影、血雾、弹道、电弧和异常扭曲；
- **情绪弧线：**设施秩序 → 运转压力 → 故障污染 → 收容失效 → Boss 高压。

### 3.2 明确排除

- 不使用 3D 模型、3D 渲染截图或伪 3D 素材作为正式画面；
- 不改成等距地图，不引入菱形 tile、等距碰撞或等距寻路；
- 不采用 Q 版、可爱化或夸张大头比例；
- 不把 AI 生成的整张“假游戏截图”当作实现规格或验收证据；
- 不直接混用风格不一致的现成素材包；
- 不彻底删除程序化纹理和动态效果。

## 4. 二维像素生产规格

### 4.1 透视与比例

游戏继续使用直角俯视世界坐标。地板、移动、碰撞和射程保持现有正交语义。角色 sprite 内部使用 3/4 绘制角度，使玩家能看见头部、躯干、手持武器和服装层次，但不改变摄像机或地图透视。

角色采用严肃写实比例。为保证 960×540 视口和高敌人数下的可读性，可以适度强化头部、武器、肩部和关键装备轮廓，但不得形成 Q 版效果。

| 类别 | 基准规格 | 说明 |
|---|---:|---|
| 地面 tile | `32×32` | 直角网格，可无缝拼接 |
| 墙体、门与大型设备 | `32×32` 模块组合或 `64×64` | 允许正面高度，但不得改变碰撞语义 |
| 玩家 | `48×48` sprite 画布 | 透明留白允许武器和动作越界 |
| 普通敌人 | `48×48` | 与玩家共享像素密度和地面尺度 |
| 精英敌人 | `64×64` | 通过轮廓和体积区分，不依赖单一颜色 |
| SCP-049 | 约 `64×80` | 高瘦、长外套、鸟嘴面具和手杖形成独特剪影 |
| 拾取物与武器图标 | `32×32` 或 `64×64` | 游戏内实体与 UI 图标可使用不同导出尺寸 |
| VFX 帧 | `64×64` 至 `128×128` | 与程序化粒子和混合模式组合 |

素材以明确像素网格绘制，使用有限色板和硬边像素簇。运行时必须验证 nearest filtering、整数缩放和摄像机运动下的清晰度；若需要修改受保护的渲染入口，必须在实施计划中单独列出原因、影响和回退方法。

### 4.2 动画分阶段交付

第一道门禁只验证静态正式 sprite 的造型、比例、地面接触点和实际游戏尺寸。静态造型通过后才进入动画生产，避免在错误造型上投入大量帧。

第一批动画目标：

- 玩家：四方向待机、移动、受击和死亡；
- 感染职员：四方向移动、受击和死亡；
- SCP-049：四方向待机/移动、狂乱暴露、受击和死亡；
- 每个循环动画使用满足动作可读性的最少帧数，不为流畅度盲目增加纹理预算；
- 武器开火、命中和异常效果优先由独立 VFX 表达，不把所有反馈烘焙进角色帧。

## 5. 首个正式素材垂直切片

垂直切片只覆盖一个代表性战斗空间和三类关键角色，不一次重做全部内容。

### 5.1 必须交付

1. **一间基金会设施战斗房间**
   - 地板、墙体、门、管道、控制设备和通风/排水结构；
   - 警示线、设施编号、血迹、污染和破损贴花；
   - 正常照明、停电和红色警报三种表现状态。
2. **玩家正式 sprite**
   - 战术人员身份、武器持握和方向轮廓清晰；
   - 与现有碰撞体、移动和自动攻击保持兼容。
3. **感染职员正式 sprite**
   - 保留“原设施人员被感染”的可识别服装痕迹；
   - 多单位同屏时仍能快速识别。
4. **SCP-049 正式 sprite**
   - 高瘦剪影、鸟嘴面具、长外套和狂乱暴露状态；
   - 普通状态、狂乱和 dying 在视觉上可区分。
5. **正式 HUD 表皮**
   - 生命、武器、时间、阶段、警报和 Boss 生命使用统一组件；
   - 使用高清面板、图标和 9-slice 兼容边框，不使用像素字体；
   - 保持战斗中心区域可见。
6. **最小动态反馈**
   - 角色脚底阴影、应急灯、命中、血雾、狂乱警报和异常扰动；
   - 保留程序化弹道、电弧和 fallback。

### 5.2 暂不进入第一切片

- 其余普通敌人和全部精英的完整正式素材；
- 所有升级、拾取物和武器的完整图标库；
- 多地图、剧情档案、角色选择或新玩法系统；
- 完整背景音乐和全部音效替换；
- 主 Scene 拆分、渲染技术栈替换或 3D 化。

## 6. 素材生产流程

### 6.1 AI 的允许用途

AI 可以用于情绪板、角色概念、服装与设备探索、材质草案和构图参考。AI 输出不得直接因为“看起来漂亮”就进入正式 manifest。

正式 sprite 必须经过：

1. 统一透视和地面接触点；
2. 统一像素密度、色板、轮廓和光源；
3. 清理透明边缘、重复噪声和生成缺陷；
4. 校正角色身份、手脚、武器和设备结构；
5. 在 Phaser 真实视口内验证尺寸和可读性；
6. 制作并检查连续动画，避免抖动、变脸、服装漂移和轮廓跳变。

AI 生成记录必须包含工具/模型、日期、原始提示、人工修改说明和准入结论。不能确认权利状态或商业使用条件的输出不得进入生产资源。

### 6.2 生产阶段

```text
风格与色板基线
  → 静态角色/环境母版
  → Phaser 实机尺寸验证
  → 方向与动画帧
  → atlas/spritesheet 整理
  → manifest 接入
  → fallback 与生命周期验证
  → 视觉 smoke 和性能检查
```

每个阶段都允许回退。静态母版未通过时不得提前批量生产动画；垂直切片未通过时不得扩展完整素材库。

## 7. 现有架构接入

### 7.1 稳定 key 与加载优先级

`src/assets/manifest.js` 继续作为正式 key 和路径的单一事实来源：

- 静态正式纹理进入 `IMAGE_ASSETS`；
- 动画角色进入 `SPRITESHEET_ASSETS` 或 `ATLAS_ASSETS`；
- `PreloadScene` 先加载正式资源；
- `fallbackTextureFactory` 只为不存在的稳定 key 生成程序化纹理；
- Gameplay 继续引用稳定 `TEXTURES` key，不在业务代码中散落文件路径。

现有 player、infectedStaff 和 SCP-049 key 不得随意重命名。若动画需要新增 atlas 动画 key，应保持静态 fallback 路径仍可启动，并在旧资源缺失时安全降级。

### 7.2 表现层与玩法层边界

素材接入可以读取现有状态并改变：

- texture、frame、animation、tint、alpha、depth、shadow 和 UI skin；
- 只读的 normal/frenzy/dying、停电、低生命和警报状态表现；
- 不影响碰撞的装饰对象和短生命周期 VFX。

素材接入不得改变：

- 伤害、生命、移速、射程、刷怪、AI、复制、升级概率、胜负和存档；
- 六分钟时间轴、Boss 状态 deadline 或敌人上限；
- Scene restart、manager API 和 localStorage schema。

## 8. 性能与资源预算

- 同一种敌人的所有实例复用同一 texture/atlas，不为单个实例生成大纹理；
- 动画帧数量由动作可读性决定，避免无意义高帧率；
- 环境贴花、阴影和警报灯采用可复用资源或轻量程序化对象；
- 临时 VFX 必须有 owner，并在结束、shutdown 和 restart 时清理；
- atlas 尺寸、纹理数量、draw call 和 230 敌人场景的帧时间必须在实施阶段记录；
- 960×540 是第一验收视口，不能通过缩小角色或隐藏 HUD 规避性能问题。

230 是敌人预算上限，不等于视觉性能自动通过。垂直切片必须在高密度场景中验证清晰度和帧率，再决定动画是否默认全量播放、降频或按距离降级。

## 9. UI 与情绪弧线

UI 保留第一轮建立的基金会专业信息结构，但不再停留在纯色矩形：

- 使用有材质层次的高清面板、边角、图标槽和状态灯；
- 主要信息稳定，异常和警报只在边界内扰动；
- 低生命、停电、狂乱和失败同时通过颜色、形状、运动或声音表达；
- 标题、武器选择、HUD、升级、暂停和结算最终共享组件语言；
- 菜单可以使用档案编号、设施标签和授权语言，但不能堆砌不可读装饰。

画面随时间线逐步失稳：开局保持设施秩序；中期增加故障、污染和局部警报；后期加强异常干扰；Boss 阶段进入明确的收容失效状态。表现层只观察既有游戏状态，不创造新的玩法状态。

## 10. 验收与证据

### 10.1 必须通过

- 在 960×540 实际游戏画面中，一眼识别基金会设施、战术人员、感染职员和 SCP-049；
- 玩家、感染职员和 SCP-049 在移动、受击和高密度场景中保持轮廓可读；
- 正常照明、停电、低生命、狂乱、失败和 restart 状态表现正确；
- 标题、武器选择、HUD、暂停、失败和 restart 使用一致的视觉语言；
- 正式资源加载时没有缺失 key、重复 key 或 atlas/frame 错误；
- 删除或故意漏掉正式资源时，程序化 fallback 仍能启动并完成基本流程；
- 现有 Node 测试、生产构建、diff check 和控制台检查继续通过；
- 视觉 smoke 使用当前 HEAD 的真实 Phaser 运行画面，不使用生成式概念图替代。

### 10.2 “成品感”门禁

垂直切片必须与当前 main 的同场景截图并排比较。若不看说明仍无法立即判断角色、环境、材质、光影和 UI 已经换代，则该切片不通过，不能进入第二批量产。

验收截图至少包含：

1. 标题或武器选择；
2. 正常战斗房间；
3. 停电或低生命；
4. SCP-049 普通状态；
5. 狂乱状态；
6. 失败或胜利结算；
7. restart 后的稳定画面。

## 11. 风险与回退

| 风险 | 缓解与回退 |
|---|---|
| AI 素材风格漂移 | 固定母版、色板和比例；人工像素清理；未通过静态门禁不制作动画 |
| 3/4 sprite 被误做成等距或 3D | 地图始终使用直角网格；只允许 sprite 内部表现 3/4 角度 |
| 正式资源破坏 fallback | 保持稳定 key；正式资源先加载；fallback 继续以 `textures.exists` 守卫 |
| 230 敌人动画导致性能下降 | 复用 atlas、限制帧、距离降级、记录高密度帧时间 |
| UI 装饰遮挡战斗 | 保持战斗中心安全区；用实机截图而非静态设计稿验收 |
| 素材许可不明确 | 每项资源进入 manifest 前完成来源和商业使用登记 |
| 动画投入后发现造型错误 | 静态 sprite 实机门禁先于动画生产 |

回退时只移除 manifest 中的正式资源条目和对应表现接入，稳定 texture key、fallback、玩法逻辑和存档保持不变。

## 12. 实施顺序

1. 建立色板、比例、透视、命名和素材登记模板；
2. 制作设施房间的静态 tile 与贴花母版；
3. 制作玩家、感染职员和 SCP-049 静态 sprite；
4. 接入 manifest 并完成真实尺寸/碰撞/可读性验证；
5. 制作角色最小动画集；
6. 接入动态光影、阴影和关键 VFX；
7. 制作高清 HUD 表皮和关键图标；
8. 完成正常、停电、低生命、狂乱、结算和 restart smoke；
9. 完成 fallback、性能、许可证和整分支审查；
10. 垂直切片通过后，再规划其余敌人、精英、拾取物和完整音频。

本设计完成的是可验证的正式素材路线，不授权一次性生成全部美术，也不把概念图质量当作游戏内质量。
