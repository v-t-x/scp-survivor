# v1.6.0「收容区焕新」集成与发布设计

## 1. 目标

把 `feature/ui-art-overhaul` 中已经实现并经过自动化验证的生产级 UI/美术纵切，按精确提交边界集成到当前稳定 `main`，形成面向普通玩家的 `v1.6.0 — 收容区焕新`。

本轮同时修复一个由表现层改造引入的冲刺方向回归，校正版本元数据、当前事实文档和素材许可说明，并建立自动化验证、浏览器 smoke、项目所有者视觉验收、PR、标签和 GitHub Release 之间的明确门禁。

## 6. 冲刺方向回归

UI cutoff 把无移动输入时的冲刺 fallback 从既有 `playerFacingAngle` 分离为最后移动方向，造成一个窄但真实的玩法变化。

v1.6 必须保持 v1.5 语义：

1. 玩家最后向上移动；
2. 玩家停止移动；
3. 霰弹枪向右成功攻击并提交朝向；
4. 玩家不按 WASD，直接触发冲刺；
5. 冲刺必须向右，而不是向上。

修复必须保留 UI 动画所需的移动方向记忆，但冲刺 fallback 继续使用 v1.5 的当前朝向语义。需要新增回归测试覆盖上述顺序，并确认有方向输入时的冲刺、其他武器、角色动画朝向和物理参数没有变化。

### 7.2 许可文件结构

许可修订的目的，是校正现有“整个仓库只有 MIT”这一过度简化的公开表达；它不应抹去已发布代码的 MIT 权利，也不应声称提供法律意见。

候选分支先按以下结构准备可审查文本；这是一套待 Level 2 许可审查确认的候选方案，不是 Agent 已经作出的法律结论：

- 顶层 `LICENSE`：许可范围总览，明确不同材料适用的许可并链接完整文本；
- `LICENSES/MIT.txt`：保留现有 MIT 文本，适用于未另行标注的独立软件代码；
- `LICENSES/CC-BY-SA-3.0.txt`：随发行提供 CC BY-SA 3.0 完整许可文本；
- `ATTRIBUTION.md`：提供 SCP Wiki、SCP-049 和项目素材的公开署名与来源；
- `docs/licensing-and-commercialization.md`：解释代码、SCP 衍生内容、项目生成素材、未来商业化复核和历史开放版本之间的边界。

顶层说明必须明确：

- 未另行标注的独立软件代码继续按 MIT 提供；
- SCP 世界观、SCP-049 以及基于 SCP 的创意表达和整合发行必须遵守 CC BY-SA 3.0 的署名与相同方式共享要求；
- SCP-049 署名为 Gabriel Jade，重写者为 djkaktus 与 Gabriel Jade，并链接原条目和 SCP Wiki；
- 项目没有使用 SCP-049 条目原图，不能把条目图片作者误写成项目生成图作者；
- 不能加入“禁止商业使用”等与 CC BY-SA 冲突的额外限制；
- 在任何公开 push、merge、tag 或 Release 前，项目所有者必须根据 Level 2 审查结果确认最终结构；若精确的软件/内容许可组合仍有实质疑问，应先咨询 SCP Licensing Team 或合格法律顾问。

### 7.3 生成素材与登记

- `docs/art/asset-register.md` 明确 53 个生产 PNG、51 个运行时加载素材和两个非运行时保留素材；
- 为 `infected-opening-sheet` 补充非运行时保留原因，避免它看起来像漏登记的正式资源；
- 公开登记表不得暴露 `%USERPROFILE%\...` 所代表的账户专属绝对路径；保留文件名、生成记录标识和 SHA-256 时，应把路径改写为仓库无关的本地归档标识；
- 指向 `.superpowers/` 的来源记录必须明确标注为“不随 Release 分发的本地审计材料”，运行时、公开署名和发布合规均不得依赖该未跟踪目录；
- 记录生成素材由项目定制生成、没有第三方图像输入，并保留提示、处理和尺寸等溯源信息；
- 可以记录 OpenAI 条款中“用户与 OpenAI 之间原则上由用户拥有输出”的事实，同时必须保留输出不保证唯一或不侵权、仍需人工审核的风险说明；
- 项目所有者的许可选择和公开分发授权是必要项目门禁，但不能被描述为专业法律意见，也不能扩写为“所有素材已经完成商业发行法律审查”。

由于许可策略属于 Level 2 决策，候选分支可以在本地准备上述文本，但在首次公开 push 前必须先生成不超过 800 字的外部 GPT 审查包，供项目所有者取得第二意见；外部 GPT 不能代替 SCP Licensing Team 或法律顾问。在 merge、tag 或公开 Release 前必须完成项目所有者的逐项许可审查；仍存在实质疑问时不继续公开发布，由项目所有者决定是否咨询专业方。

## 8. 验证与视觉验收

### 8.1 自动化门禁

候选分支完成合并和修订后，至少执行并保存结果：

1. `npm ci`；
2. `npm test`，其底层运行全部 Node 测试并包括新增冲刺回归测试；
3. `python scripts/art/test_pixel_tools.py`；
4. `npm run build`；
5. Markdown 本地链接检查；
6. `git diff --check`；
7. 版本号、`DEBUG_MODE=false`、cutoff 祖先关系和 cutoff 后 4 个提交未进入候选分支的检查；
8. 运行时 manifest 与 asset register 覆盖关系检查；
9. 最终 Git 状态和未知文件检查。

Vite 的大 chunk 警告可以作为已知技术项记录，但不得隐藏。若出现新的构建错误、测试失败、缺失素材、控制台错误或未知工作树改动，则候选版本不通过。

实施计划必须给出可复制执行的 Markdown 本地链接检查命令、失败条件，以及在最终 `DEBUG_MODE=false` 的干净候选版本上到达六分钟终局、胜利和失败状态的方法；不得用已提交的调试开关替代最终构建 smoke。

### 8.2 源码审查

自动化通过后，调用独立源码审查，至少覆盖：

- cutoff 的集成边界和 cutoff 后 4 个提交确实未进入候选分支；
- Scene shutdown/restart、manifest/fallback、物理 body 和战斗通知链；
- UI/Art 是否暗中改变数值、刷怪、AI、存档、奖励、胜负或六分钟流程；
- 冲刺回归修复及其测试是否真正保留 v1.5 语义；
- 许可、来源、README、设计事实和 Release 文案是否一致；
- 未跟踪 `.superpowers/` 与本机绝对路径没有进入提交或公开文档。

审查发现的问题必须在候选分支修复，重新运行全部自动化并再次审查最终 HEAD。只有最终候选 HEAD 获得源码审查批准，才能进入浏览器与项目所有者验收。

### 8.3 浏览器 smoke

自动化通过后，启动一个由该 worktree 所有、固定端口、可跨对话存活的本地 Vite 服务。启动前检查端口占用；不得为了抢端口终止未知进程。交付链接前再次确认进程归属和 HTTP 可用性。

Agent 先完成：

- 960×540 标准画面检查；
- 一个不同浏览器 viewport 的布局和点击区域检查；
- 标题到武器选择再到任务的完整进入流程；
- 控制台、资源请求和 fallback 警告检查；
- 暂停、TAB、升级、失败、胜利和至少两次 restart；
- 断电、精英预警、R-17 家族、SCP-049、战斗反馈和 Boss 表现；
- 有输入与无输入冲刺方向回归。

### 8.4 项目所有者验收

Agent smoke 不能代替项目所有者。必须把稳定本地链接和简短清单交给项目所有者，由其确认：

- 标题和武器选择清晰；
- 玩家、敌人、Boss 与设施比例和风格可接受；
- HUD、覆盖层、警告和战斗反馈没有明显遮挡或视觉冲突；
- 鼠标热区、暂停、TAB、升级、结果页和 restart 可用；
- 没有明显缺图、fallback 泄漏、控制台错误或无法接受的美术问题；
- v1.5 的冲刺、战斗、奖励、胜负和六分钟流程语义仍然成立。

只有项目所有者明确给出 v1.6 视觉/试玩通过结论，才能把候选版本称为可进入 GitHub 集成阶段。

## 9. GitHub 集成与发布顺序

外部操作按以下顺序执行，每个授权点都基于当时的最新证据：

1. 本地候选分支完成全部自动化、最终 HEAD 源码审查、浏览器 smoke、许可审查和项目所有者视觉验收；
2. 展示最终 diff、提交列表、验证结果、许可摘要和 Git 状态；
3. 取得明确授权后 push `codex/release-v1.6` 并创建新的 Draft PR；
4. 检查 PR 最新 HEAD 的 GitHub Actions，CI 必须包含测试与构建；
5. 若 CI、review 或用户反馈导致修改，修复后重新执行全部 final-head 门禁；任何玩家可见修改还必须重新取得项目所有者视觉验收；
6. 再次展示 PR、目标 commit 和所有门禁，取得明确授权后把 PR 标记 ready 并 merge；
7. 在最新 `main` 上重新 fetch、核对 ancestry、运行最终测试/构建/smoke 并确认远端同步；
8. 展示精确 tag 目标，取得标签授权后创建并推送 `v1.6.0` 标签；
9. 再次展示最终 Release 名称、正文和 tag 目标，取得发布授权后创建非草稿、非预发布的 GitHub Release，并把它设为 Latest；
10. 通过 GitHub API 反查名称、正文、标签目标、草稿/预发布/Latest 状态；
11. 最终报告 `main`、远端、标签、Release、PR、worktree 和已知风险状态。

GitHub Release 只使用标签自动附带的 Web 源码归档，不附加 Windows 安装包或未经独立验证的二进制产物。在线 Demo 只有在被核对为同一最终提交时才能称为 v1.6；否则继续标记为未核对或独立部署。

## 10. 失败处理与回退

- 远端漂移、ancestry 异常、未知工作树修改或意外冲突：停止合并/发布操作，保留现场并重新审计；
- 自动化失败：在候选分支定位并修复，随后重跑全部 final-head 门禁；
- 视觉验收失败：保持 Draft/本地候选状态，不 merge、不 tag、不发布；
- 许可审查未通过：允许保留候选代码和审查材料，但不得公开声明版本许可就绪；
- push 成功但 PR 创建失败：保留远端分支，修复问题后重试 PR，不删除分支或重写历史；
- merge 成功但 tag/Release 失败：不得回滚或重写已经公开的 main；修复剩余步骤并准确报告部分完成状态；
- Release 创建后发现文案错误：只更新 Release 文案或新增修正文档提交，不移动既有 tag；
- 不自动删除 UI 分支、PR #3、release 分支或 worktree。清理必须作为发布后的独立授权操作。

## 11. 完成标准

v1.6 只有同时满足以下条件才算真正发布完成：

- `main` 包含精确 cutoff 的完整 UI/美术实现，不包含 cutoff 后 4 个角色规划提交；
- 冲刺行为通过新增回归测试并保持 v1.5 语义；
- 正式素材、manifest、fallback、文档和版本元数据一致；
- 顶层许可总览、完整许可文本、SCP/SCP-049 署名和资产登记一致；
- Node、Python、构建、链接、diff、浏览器和 restart 门禁通过；
- 最终候选 HEAD 获得独立源码审查批准，且审查后没有未经复核的代码改动；
