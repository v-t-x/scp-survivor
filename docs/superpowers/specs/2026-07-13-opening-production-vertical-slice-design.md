# 开局成品级视觉竖切片设计

## 1. 目标

本阶段把玩家从标题页进入游戏后的前 60 秒重制为一段视觉统一、可用于宣传截图和后续美术生产定标的成品级竖切片。

本阶段不是继续在旧界面上替换颜色或零散贴图。标题页、武器选择、战斗 HUD、初始设施房间、玩家与首批敌人必须作为同一套视觉系统重新设计，使玩家能够立即识别基金会设施、收容失效和战术生存主题。

成功标准：不了解开发历史的玩家看到标题、武器选择和开局战斗三张截图时，应认为它们来自同一款接近正式制作阶段的游戏，而不是浏览器原型或素材拼贴。

## 2. 已确认方向

- 世界表现采用精细像素 2D 俯视角，不使用 3D、等距 3D、Q 版或写实高清角色。
- UI 采用高清战术界面，与像素世界形成清晰分层。
- 核心气质为“基金会工业设施的专业压迫感”逐步转向“异常失控的恐怖感”。
- 保留 Phaser 3、Vite、现有 Scene 生命周期、Preload/manifest/fallback、AudioManager、UIManager 和存档合同。
- 不改变伤害、敌人 AI、刷怪、碰撞、升级概率、六分钟时间线、胜负或存档语义。
- 地图物件必须具有空间或叙事含义，不允许为了填空而随机散布门、通风口、控制台或污渍。

## 3. 范围

### 3.1 标题页

标题页表现为设施安保终端前的收容失效现场，而不是纯色背景上的文字和矩形按钮。

- 左侧承担游戏标题、设施编号、警戒等级与一句核心任务说明。
- 右侧设置半开放安保门、监控屏、状态灯和局部设施结构，形成明确场景焦点。
- 主入口“开始行动”使用终端操作组件，包括状态边框、确认反馈和装载状态。
- 永久加成、设置和辅助信息作为次级入口，不能与主入口争夺层级。
- 背景允许低频警报灯、监控噪点和远处阴影动画，但不得影响文字可读性。

### 3.2 武器选择

武器选择表现为基金会军械配置界面，不再使用网页式三张空白卡片。

- 三件主武器分别进入独立军械槽，展示 96×96 武器插画、名称、战术定位和关键机制。
- 信息优先描述玩法差异：射程、射速或装填、控制能力与适用场景；避免堆积次要数值。
- 选中状态包含装备锁定框、状态灯、确认按钮变化和短促装配反馈。
- 未选中、悬停、选中、不可确认四种状态必须视觉可辨。
- 三种武器继续使用现有 ID、数值和选择流程。

### 3.3 初始设施战斗区

开局区域重构为横向收容维护大厅。中心保留至少半径 260px 的清晰战斗空间，边缘由具有空间关系的设施模块围合。

- 房间必须有可识别的入口、墙体连续关系、维护通道、观察区和设备区。
- 地面由主地板、维护通道、危险标线、磨损或异常污染组成；每层都有明确用途。
- 门必须连接墙段或通道，控制台必须靠近门、观察窗或设备，通风口必须依附墙面或维护区域。
- 血迹、破损和异常污染用于讲述事件路径，不能均匀或随机填满地面。
- 所有新增结构在本阶段保持纯视觉，不新增碰撞体或改变可通行区域。
- 初始相机截图内必须同时看到玩家、房间边界关系、至少一个明确入口和至少一个功能设备区。

### 3.4 战斗 HUD 与任务呈现

HUD 从“左上角文字堆”改为分区战术界面。

- 左上：当前任务模块，仅展示当前阶段目标、下一节点或关键进度。
- 左下：生命、护甲或生存状态；本阶段没有护甲玩法时不得伪造护甲数值。
- 右下：主武器、弹药或冷却、闪避状态。
- 顶部：设施状态条；正常时保持收缩，停电、封锁、异常物品和 Boss 事件时展开。
- 次要调试和说明文字不得长期占据主要战斗区域。
- HUD 只重新组织现有信息，不新增玩法状态或改变信息含义。

### 3.5 角色与敌人

首批角色至少覆盖玩家、感染职员和开局 60 秒内实际出现的其他普通敌人，不能让其中任何一种退回简单几何图形。

- 玩家与普通敌人使用 48×48 四方向 spritesheet。
- 每个首批角色至少具备待机 4 帧、移动 6 帧、受击 2 帧。
- 动画只驱动显示，不改变 physics body、移动速度、攻击窗口或 AI 状态机。
- 角色朝向可由现有移动向量和攻击方向映射；缺失方向时允许镜像，但必须在登记表记录。

## 4. 素材规格

| 类型 | 规格 | 要求 |
|---|---:|---|
| 基础地面 tile | 32×32 | 无缝、硬边像素、受控调色板 |
| 墙体与设施模块 | 64×64 / 96×96 | 模块化连接，透明背景，不伪造碰撞 |
| 玩家与普通敌人 | 48×48/帧 | 四方向，待机/移动/受击 |
| 武器主插画 | 96×96 | 透明背景，轮廓和功能结构清楚 |
| 任务与状态图标 | 24×24 / 32×32 | 在 100% 缩放下可辨 |
| UI 面板 | 九宫格或可伸缩组件 | 高清边缘，统一间距和状态层级 |
| 按钮 | 高度 48–64px | 默认/悬停/按下/禁用状态完整 |

像素素材使用 nearest filtering，不允许混入平滑缩放、伪 3D 透视或风格不一致的高频细节。正式素材继续通过 manifest 加载，并保留程序化 fallback。

## 5. 视觉系统

### 5.1 色彩

- 基础：低饱和钢蓝、石墨灰、冷白。
- 基金会正常状态：低亮青绿。
- 警告与任务强调：克制的琥珀黄。
- 收容失效与生命危险：深红，不允许整屏长期纯红覆盖。
- 异常污染：暗紫红或病态灰绿，仅用于异常对象和事件路径。

### 5.2 层级

世界层依次为地面、结构、设备与痕迹、角色、攻击/VFX。HUD 独立于世界层，任务、设施警报和结算使用固定优先级，不得互相覆盖。

### 5.3 动效

动效服务于状态沟通：按钮确认、装备锁定、受击、设施警报和异常事件。禁止用持续抖动、扫描线或闪烁掩盖素材与布局不足。

## 6. 代码与组件边界

- Theme 继续作为颜色、字体、间距和状态 token 的唯一来源，扩展 token 时不得复制硬编码颜色。
- 菜单、HUD、房间构图和角色动画分别放入清晰模块；不得继续把整屏视觉逻辑堆入 `menus.js`、`hud.js` 或 `world.js`。
- 房间布局使用语义化模块描述，例如 `entryDoor`、`observationBay`、`maintenanceConsole`，而不是无含义的坐标数组。
- 角色的 physics proxy 保持现状；动画 sprite 或 presentation helper 只同步位置、朝向、帧和显示状态。
- Scene restart、SHUTDOWN、DESTROY 必须清理新增容器、动画与事件监听。
- 正式资源缺失或加载失败时，必须回落到稳定 key 的 fallback，不阻断开局流程。

## 7. 数据流

现有玩法状态继续是唯一事实源：

1. Scene 和现有 mixin 产生生命、武器、任务阶段、设施事件和 Boss 状态。
2. presentation adapter 将这些状态转换为 UI 文案、可见性、动画状态和主题色。
3. UI/Art 组件只读取转换后的展示状态，不写回伤害、时间线或 AI 数据。
4. restart 或 shutdown 清空展示缓存，下一局重新从 Scene 状态构建。

## 8. 错误处理与回退

- 每个正式 texture key 必须有 existence-guarded fallback。
- spritesheet 缺失时允许使用单帧 fallback，但必须保留原 physics body 与玩法状态。
- UI 图标缺失时显示统一的缺失图标，不得显示空白卡片或破损 texture。
- 动画定义找不到帧时降级为静态帧并记录一次警告，不能在每帧重复输出。
- 停电、暂停、失败和重启后，所有 tint、alpha、动画和临时 UI 状态必须恢复。

## 9. 验收标准

### 9.1 视觉验收

- 标题页与当前版本在构图、按钮造型、背景场景和信息层级上有明显差异。
- 武器选择不再呈现为三张网页卡片，三件武器在缩略截图中仍可区分。
- 初始战斗截图能够读出房间入口、墙体关系、设备区和中心战斗区。
- 地图中不存在无法解释用途或依附关系的散落设施贴图。
- 玩家和首批敌人不会以静止贴纸滑行；移动与受击状态肉眼可辨。
- HUD 不再把任务、武器、生命和阶段说明堆在同一文字区域。

### 9.2 行为验收

- 标题进入、武器选择、开始战斗、暂停、返回标题、失败重启路径全部可用。
- 角色动画前后 physics body 的尺寸、半径和 offset 不变。
- 正常、停电、低生命、Boss 出现、失败和重启后无残留 tint 或 UI。
- 控制台无 texture、animation、listener 或 restart 相关错误。

### 9.3 工程验收

- 素材尺寸、格式、调色板、帧数、manifest key、fallback 和许可证登记均有自动测试。
- `node --test`、`npm run build` 和 `git diff --check` 通过。
- 960×540 至少保存标题、武器选择、正常战斗、暂停、停电、Boss、失败和重启证据。

## 10. 非目标

- 不在本阶段重做完整六分钟内所有敌人、事件和升级选项。
- 不在本阶段重做 SCP-049 的完整动画集；只确保既有 Boss 表现与新版房间、HUD 和状态清理兼容。
- 不新增地图碰撞、房间切换、探索任务或完整关卡系统。
- 不改变 Boss 机制、胜负条件、数值平衡或存档。
- 不一次性替换全部音频、VFX 和程序化 fallback。
- 不开展 App Platform、发行、商业化或许可证策略重定。

## 11. 实施顺序

1. 建立新版视觉合同、组件边界与截图基线。
2. 重做标题页和通用按钮/面板系统。
3. 重做武器选择与军械槽。
4. 建立语义化初始设施房间和地面层。
5. 接入玩家与首批敌人 spritesheet 和显示状态适配。
6. 重组战斗 HUD 与任务模块。
7. 补齐停电、暂停、失败、重启和 Boss 状态验证。
8. 进行整屏视觉审查；未达到截图标准时不进入下一批内容扩展。
