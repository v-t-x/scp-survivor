# 实战画面成品化垂直切片设计

## 1. 决策与目标

本阶段采用“实战成品化垂直切片”方案，目标是让玩家从标题页、军械库进入任务后的第一分钟仍然感受到同一款正式游戏，而不是从完成度较高的菜单退回原型战场。

验收重点是固定 960×540 视口下的实际游玩画面：设施应具有明确空间关系，HUD 应突出战斗信息，暂停、升级和结算应使用统一的基金会终端语言，人物与敌人应通过光影和反馈稳定地落在场景中。

本阶段沿用已确认的精细 2D 像素风格，不切换为 3D、预渲染 3D 或高清手绘混合风格。标题页和军械库已建立的基金会工业终端语言作为质量与风格基线，不在本阶段重新设计。

## 2. 范围与等级

本设计实现既有《UI、美术、音频与资源方向》，不改变项目视觉路线、玩法循环或技术栈，按 Level 1 表现层垂直切片处理。实施可分批提交，但所有批次共同构成一个可完整验收的实战切片。

包含：

- 设施环境构图、正式环境素材和只读环境灯光；
- 战斗 HUD 的层级与布局收紧；
- 升级、暂停、当前构筑、失败和胜利覆盖层统一；
- 角色接触阴影、射击、命中、死亡和警报反馈；
- 相应 manifest、fallback、资源登记、测试和浏览器视觉验收。

不包含：

- 伤害、生命、减伤、武器行为、AI、刷怪、升级概率、任务、胜负或六分钟时间轴修改；
- 玩家、R-17 敌人或 SCP-049 的碰撞体和物理几何修改；
- Gameplay Scene 拆分、Preload 合同重写或 manager 公共接口重构；
- 新地图、新任务、新异常机制或新的局外成长；
- 重新设计标题页、军械库或已确认的七种 R-17 spritesheet；
- 恢复悬浮武器、精英头衔、黄色常驻圆圈或通用稀有度边框。

## 3. 视觉总则

### 3.1 像素素材规格

- 使用顶视角精细像素素材和最近邻过滤；
- 环境资源采用 16 px 基础网格，模块尺寸使用 16 的整数倍，优先使用 64、96 和 128 px 模块打破小纹理无限平铺；
- 保留现有玩家和敌人 spritesheet 帧尺寸、帧数、方向与缩放合同，不通过本阶段改变单位物理或显示尺度；
- 同一批资源共享左上方冷光、2–3 px 关键轮廓、低饱和钢灰主体和受控的青、黄、红状态色；
- 战场背景的明度和纹理对比低于角色、敌人、子弹和功能预警；
- 禁止引入 3D 透视不一致、照片纹理、模糊缩放或与现有武器/敌人不同的渲染密度。

### 3.2 正式素材与程序化表现边界

必须使用正式像素素材：

- 大块地面、服务地板、墙体、门、观察窗、设备和污染痕迹；
- 升级与异常突变图标；
- 事故报告、重新收容确认等印章和终端表面装饰。

继续程序化生成：

- 生命、经验、弹药和冷却条；
- 动态遮罩、弹道、攻击范围、护甲弧和危险预警；
- 火花、闪电、命中碎片和短暂粒子；
- 素材缺失时的开发 fallback。

动态框架可以继续使用 Phaser Graphics 保持尺寸和状态响应，但不得再以纯矩形和换色代替完整的终端视觉；面板必须组合切角结构、正式表面装饰、状态灯、编号和清晰的排版层级。

## 4. 设施环境与画面构图

场景主题是“收容事故中的设施维护枢纽”，由三个可读区域组成：

1. **中央战斗区**：低对比、较干净的金属主地板，承担移动、敌人、弹道和掉落物的主要可读空间；
2. **左侧入口区**：闸门、安保编号、服务通道和进入方向，建立任务起点；
3. **右侧维护区**：观察窗、管线、配电柜和维修地板，形成更密集的工业轮廓。

污染痕迹从外围事故源向中央延伸。任务开始时保持克制，不超过首屏视觉面积的大约 10%，避免开局即把整个设施做成同质化血污场。

构图规则：

- 每个设备必须依附墙体、门、观察窗、配电区或维护链路，删除没有父级关系的随机装饰；
- 管道必须在视觉上连接设施，污染必须形成连续事故轨迹；
- 中央可行走区域只放置污渍、格栅、线缆、标线和投影等明确可穿过的元素；
- 不在无碰撞区域放置看似会阻挡玩家的大型设备或墙体，避免视觉承诺与物理规则冲突；
- 地板使用大块主材质、区域变体和低频磨损层，消除重复“壁纸感”；
- 墙根、设备和角色使用只读遮蔽或阴影层，不修改 Arcade Physics；
- 冷色顶灯维持中央清晰，少量红色应急灯用于外围事故源和已有警报状态；
- 停电、设施事件和 Boss 阶段只读取现有 presentation 状态，改变灯光、色调和警报层，不创建新玩法状态。

## 5. 战斗 HUD

HUD 从四个同权重黑色面板收紧为一套战术终端：

- **左上任务终端**：显示任务阶段、主要目标和下一节点时间；击杀数降为次要信息；
- **上方设施状态条**：稳定时保持单行紧凑状态，停电、设施事件或 Boss 阶段时扩展为警告条；
- **左下生命与成长**：生命为最高优先级，等级和经验紧凑排列；低生命才启用红色脉冲与边缘警告；
- **右下武器与机动**：正式武器图标、弹药/冷却、装填和闪避使用条形、数字与短标签表达；
- **右上系统快捷区**：视觉上收紧为“M 音频”与“ESC 暂停”，但保留足够的鼠标命中区域；
- **拾取范围**：取消常驻绿色圆圈，仅在范围变化、获得物品或查看构筑时短暂显示。

HUD 统一规则：

- 中文承担主要信息，英文、编号和分类码只作终端辅助；
- 灰蓝为默认，绿色只代表稳定，黄色只代表警告，红色只代表危险；
- 默认面板降低边框亮度，只有当前重要状态得到高亮；
- 文本更新不得造成面板位移或尺寸跳动；
- HUD 只消费现有只读 presentation 数据，不写入 Scene 玩法字段；
- 960×540 下人物周围保持无常驻 HUD，战场视觉仍是截图主体。

## 6. 统一终端覆盖层

暂停、升级、当前构筑、失败和胜利共享以下组件：

- 全屏暗化、轻微扫描线和边缘遮蔽；
- 切角终端面板、状态灯、编号与警示条；
- 统一的主要、次要、危险和禁用操作按钮；
- 支持图标、等级、效果和风险标识的档案卡片；
- 幂等的事件解绑和显示对象销毁。

具体界面：

### 6.1 升级

- 三个选项表现为“现场授权方案”；
- 每个现有升级拥有正式精细像素图标；
- 普通强化使用冷灰蓝档案，异常突变使用紫红污染警示条；
- 突变同时显示收益和风险，不仅依靠颜色区分；
- 选中反馈表现为终端授权确认，不使用整张卡片简单变色作为唯一反馈。

### 6.2 暂停

- 标题为“行动暂停”；
- 显示站点编号、当前任务、运行时间和设施状态；
- 主要操作是“继续行动”，危险操作是“终止并返回标题”。

### 6.3 当前构筑

- 以装备与授权档案呈现；
- 按主武器、常规强化、异常突变分组；
- 使用图标、等级和简短数值，避免连续大段文字。

### 6.4 失败与胜利

- 失败使用“事故报告 / 行动终止”结构和红色状态戳；
- 胜利使用“重新收容确认 / 任务完成”结构和稳定状态戳；
- 两者共享统计字段、排版与返回行动准备的操作区；
- 保持现有学分结算、restart、返回标题和胜负语义。

覆盖层动画只允许短淡入、扫描和状态灯变化，不增加阻碍操作的长转场。颜色不得成为唯一状态提示。

## 7. 战斗反馈

### 7.1 落地与朝向

- 玩家和敌人获得独立像素接触阴影；
- 阴影不参与碰撞，不通过主体 sprite 的缩放、旋转或物理偏移实现；
- 玩家继续面向自动攻击方向，移动不改变朝向；
- 不恢复悬浮武器；枪口火光和弹道从现有攻击朝向的角色边缘发出。

### 7.2 命中与死亡

- 普通命中采用短促闪白或材质染色，以及小型火花、组织碎片或裂隙粒子；
- 重武器可以使用轻微、短时的表现层相机冲击，但不得改变瞄准、移动或玩法计时；
- 死亡反馈按异常材质区分为组织崩解、裂隙收缩或甲壳破裂；
- 死亡效果短时存在并受数量上限控制，不产生长期遮挡。

### 7.3 功能性预警

- 甲壳门护甲弧、冲锋预警和缺帧体闪现预警保留；
- 功能预警始终高于装饰粒子和环境灯光；
- 不恢复通用精英文字、黄色轮廓或常驻类型图标。

## 8. 结构与数据边界

本阶段不拆 Scene，而是在现有 UI/Art 边界内形成四个独立单元：

1. **设施视觉层**：负责正式环境素材、分区、静态遮蔽和读取已有状态的灯光表现；
2. **战术 HUD 视图**：消费冻结或只读 HUD presentation 数据，拥有自己的显示对象和生命周期；
3. **终端覆盖层组件**：提供覆盖层、面板、按钮和档案卡片，各业务界面只提供内容与回调；
4. **战斗反馈控制器**：接收攻击、命中和死亡已经提交后的只读通知，管理可复用或有上限的短暂显示对象。

数据流必须是：

玩法状态或已提交事件 → 纯 presentation 模型/只读快照 → 视图或反馈控制器

视图返回值不得决定伤害、目标、弹药、击杀、经验或胜负。反馈控制器不存在、构造失败或运行时报错时，玩法结果和既有 fallback 必须保持一致。

## 9. 错误处理与生命周期

- 所有新资产通过现有 manifest 与 Preload 管线加载；
- 每个正式资产具有兼容 fallback；缺失或帧合同错误时仅警告一次；
- 组件构造发生中途失败时，必须销毁已经创建的显示对象后再交回 fallback，不能只清空引用；
- overlay、HUD、环境层和反馈控制器在 shutdown、destroy、restart 与返回标题路径上幂等清理；
- 高频反馈优先复用显示对象或设置活动数量上限，避免每次攻击产生不可控的创建/销毁和 GC 抖动；
- 新素材必须更新《资源登记表》，记录来源、生成方式、修改状态、商业使用和署名要求。

## 10. 实施批次

实施计划按以下顺序拆分，每批均保持可运行：

1. 环境材质规范、正式资源合同和维护枢纽构图；
2. HUD presentation 与战术 HUD 视图；
3. 共享终端组件及升级、暂停、构筑、失败、胜利界面；
4. 接触阴影、射击、命中、死亡和设施状态反馈；
5. 整体视觉审查、修复循环与固定端口人工试玩。

后一批不得依赖尚未完成的玩法接口。若确实缺少事件通知，只能提出最小、失败安全、发生在玩法结果提交之后的只读接口。

## 11. 测试与验收

### 11.1 自动门禁

- 环境布局语义关系：设备有父级、污染轨迹连续、中心安全区没有假障碍；
- 正式 PNG 的尺寸、RGBA、最近邻、限定色板和硬边像素合同；
- manifest、fallback、纹理 key 唯一性和缺失资源警告；
- HUD presentation 冻结、比率钳制、稳定布局和无玩法写入；
- overlay 按钮热区、键盘路径、升级选择、暂停恢复、返回标题、失败重启和胜利重启；
- 反馈控制器缺失、正常和抛错三种路径的玩法结果等价；
- Scene shutdown、destroy 和 restart 后没有监听器、Tween、Timer 或显示对象泄漏；
- 全量 node --test、npm run build、git diff --check 与最终 Git 状态检查。

### 11.2 视觉门禁

在 960×540 实机中检查：

- 标题 → 军械库 → 任务开始的风格连续性；
- 隐藏 HUD 后仍能读出入口、中央战斗区和维护区；
- 中央地板没有明显小纹理重复，角色、敌人、弹道和预警不被背景吞没；
- 默认、低生命、停电、设施警告和 Boss 状态的 HUD 层级；
- 升级、暂停、当前构筑、失败和胜利覆盖层；
- 不同武器的枪口、弹道、命中与死亡反馈；
- 无缺失纹理、重复动画或运行时错误。

自动门禁和独立代码/视觉审查通过后，使用只读端口检查启动固定地址 http://localhost:64126/，由项目所有者完成最终人工试玩。不得仅凭测试或静态截图宣称视觉验收完成。

## 12. 成功标准

本阶段完成时：

- 任意截取任务开始后第一分钟的战斗画面，都能明确识别为基金会设施中的正式游戏；
- 标题、军械库、战斗 HUD 和覆盖层属于同一套视觉系统；
- 地图装饰具有空间和叙事关系，不再像随机贴图；
- 玩家无需阅读长句即可识别生命、任务、武器和设施危险；
- 战斗反馈清晰但不遮挡功能性预警；
- 所有玩法、物理、存档和胜负语义与实施前一致。
