# SCP Survivor 产品愿景

> **权威范围**：产品目的、SCP 差异化原则与重大功能判断标准。本文描述“为什么做、不能偏离什么”，不直接批准具体开发任务。

## 为什么选择 SCP 世界观

项目起点是一个 2D 俯视角 Survivors-like 原型，但目标不是复刻传统割草玩法。SCP 世界观天然适合持续危机、设施压迫、未知规则与任务取舍：

- 收容失效带来的持续危机感；
- 基金会设施内部的压迫与未知；
- 每个异常实体独立的规则与应对方式；
- 安保人员、研究设施、异常物品与收容设备等鲜明题材元素；
- 停电、封锁、警报、设施故障和区域失控；
- 从单纯击杀扩展到处理异常、恢复设施、完成任务与撤离。

SCP 世界观必须影响机制、事件、敌人行为、地图目标和玩家决策，而不只是名称和美术包装。当前 SCP-049 Boss 是早期尝试，不代表 SCP 核心特色已经完整建立。

## 不能成为普通割草游戏的换皮版本

移动躲避、自动攻击、击杀、经验、升级和定时生存可以继续作为基础循环，但不能成为全部内容。项目需要逐步建立以下差异：

- SCP 实体拥有独立应对规则，而不只是更高血量；
- 设施会发生停电、封锁、警报与异常事件；
- 玩家需要在战斗、探索、事件处理和目标推进之间取舍；
- 任务可以包含收容、恢复供电、启动设备或撤离；
- 异常物品可以强力但带有代价或风险；
- 地图状态与收容失效程度会影响环境、敌人和目标；
- Boss 战体现设定和规则，而不只是弹幕与血量增长。

纯粹增加伤害、生命或刷怪数量，但没有带来新机制、决策或氛围的功能，优先级较低。

## SCP 驱动的体验支柱

1. **规则感**：异常必须有可以观察、理解和应对的规则。
2. **设施感**：地图不是空旷竞技场，而是会失效、封锁和恢复的基金会设施。
3. **任务感**：生存之外还应存在事件处理、设施操作、收容或撤离目标。
4. **取舍感**：玩家在战斗、探索、风险物品与目标推进之间做决定。
5. **压迫感**：视觉、音频和事件共同营造未知与失控，而不是只堆敌人数量。

## 功能提案的四个问题

设计新功能前依次判断：

1. 这个功能是否强化 SCP 世界观？
2. 它是否产生新的玩家决策？
3. 它是否与普通割草游戏形成差异？
4. 它是否只是增加数值和敌人数量？

前三项缺乏明确收益、第四项为“是”的提案，应降低优先级或重新设计。

## 已确认方向

- 保留当前稳定且可玩的版本作为持续回归基线；
- v2 继续强化 SCP 内容、规则、事件、Boss 与关卡差异；
- 表现层采用正式素材与程序化效果并存的混合方案；
- 程序化纹理与音频继续作为开发 fallback；
- 旧版 localStorage 数据尽量保持兼容；
- 架构与素材管理保留未来发布和商业化调整空间。

具体开发分工和里程碑以 [开发策略](./development-strategy.md) 为准。

## 尚未批准的长期设想

以下内容可以分析、比较和预留兼容性，但不是已批准任务：

- 全面改造为多个 Gameplay Scene；
- 多个完整地图、区域或可操作角色；
- 大量真实 SCP 实体及其具体名单；
- 重新加入或扩展收容节点、区域供电与电梯撤离；
- 更复杂的任务、探索、剧情、档案和叙事；
- 完整中英文 i18n；
- macOS、Linux、自动更新、Steam 或其他商业平台；
- 联机、多人合作、排行榜、账号、云存档与跨设备同步；
- 彻底替换程序化纹理或音频；
- 完整 Roguelite 局外成长；
- 未来所有代码继续采用 MIT License。

未获得项目所有者或 Project Lead 明确授权前，Agent 不得把这些设想转化为实现任务。

## 决策优先级

当范围不明确时，按以下顺序判断：

1. 保证稳定版本不被破坏；
2. 保持 SCP 世界观和玩法差异化；
3. 保持不同 Agent 的职责边界；
4. 优先完成当前已批准的里程碑；
5. 避免没有明确收益的大规模重构；
6. 保留未来扩展和商业化空间；
7. 未批准的长期设想只记录，不直接实现。

## 最终决策权

项目所有重大产品和架构决定最终由项目所有者确认。Project Lead 负责维护方向、审查方案、分配职责与控制合并顺序。
