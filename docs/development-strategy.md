# 开发策略与并行路线

> **权威范围**：当前开发路线、分支职责关系与里程碑授权边界。本文授权工作方向，不自动批准具体功能。

## 稳定版本保护

当前正式稳定基线（具体版本见[当前项目状态](./project-status.md)）必须持续可构建、可启动和可回归。新玩法、客户端封装、UI/Art 和资源改造在独立分支与 worktree 中进行；任何共享接口变更都要先评估其他分支影响。

项目继续使用 Phaser 与 Vite。旧 localStorage 数据应尽量兼容，程序化素材继续作为资源缺失时的 fallback。

## 为什么 App Platform 与 v2 并行

App Platform 负责平台、构建、安装与发布问题；v2 负责玩法和内容。两者风险不同，可以并行推进：

- 提前验证网页游戏能否稳定封装为客户端；
- 尽早发现资源路径、音频、窗口、权限、离线和打包问题；
- 在 v2 完成前积累可执行版本、安装包与发布流程经验；
- 避免平台工作隐藏在玩法大版本末尾集中爆发。

第一阶段 App 验证优先 Windows，但 App Platform 的长期职责覆盖桌面、移动端、PWA、启动器与未来客户端容器。最终发布整合等待玩法、UI 与资源接口相对稳定。

## 四条工作流

### `dev/v2`

负责 SCP、Boss、敌人、武器、事件、关卡、数值和核心玩法。已集成至 `main` 的玩法由当前基线承载，后续玩法职责仍由 `dev/v2` 承担；分支名表示职责边界，不代表 `v2.0.0` 已发布。目标是逐步摆脱普通原型感，建立 SCP 规则与任务差异。

### `dev/app-platform`

负责本游戏所有客户端形态、平台集成、安装、更新、签名、商店、离线、安全、打包和发布流程。不得隐藏修改玩法语义。

### `feature/ui-art-overhaul`

负责 UI、美术、纹理、动画、VFX、音频素材和表现层接入。不得改变玩法数值、胜负条件或敌人行为。

### `refactor/ui-foundation`

Phase 1 已完成并冻结。它提供 Preload、manifest、fallback、AudioManager 和 UIManager 等基础接口，只接受明确批准的修复。

## 合并与发布原则

- `main` 是受保护的稳定集成基线；
- 分支必须经过计划审查、diff 审查、构建与相关 smoke test；
- 共享文件变更由 Project Lead 指定执行者和合并顺序；
- 不因单个分支“看起来完成”而直接发布；
- 客户端最终发布整合等待玩法、UI 和资源系统达到相应里程碑；
- 正式发布、签名、商店提交和商业发行需要单独授权。

## 已批准的近期方向

- 保持稳定版本运行；
- v2 继续强化 SCP 玩法差异；
- App Platform 先验证 Windows 客户端封装；
- UI/Art 分阶段引入正式素材并保留程序化 fallback；
- 所有开发继续使用 Git 分支和 worktree 隔离；
- Agent 按 [治理规则](../AGENTS.md) 和分支手册工作。

## 未批准事项的处理方式

[产品愿景中的长期设想](./product-vision.md#尚未批准的长期设想)只能分析、提出选项或记录兼容性。它们不因出现在文档、历史 Roadmap 或设计计划中而获得实施授权。具体功能必须有直接指令或经过批准的实施计划。
